AWS Account & IAM Setup

Create your AWS account, secure it with IAM, and install the CLI on your Windows machine

ℹ️
New to AWS? Start here. This section takes ~20 minutes. You will never use the root account after setup — we create a dedicated deploy user for everything.
1 Create AWS Account at aws.amazon.com ▾
  • Go to aws.amazon.com → click Create an AWS Account
  • Use a dedicated email like aws@orbitumai.com — not personal email
  • Choose Personal account type, enter credit card (free tier covers most of this)
  • Select Basic Support Plan (free)
  • After signup, go to IAM → Account Settings → Enable MFA on root account immediately
⚠️
Critical: After creating the root account, you will only use it to create one IAM admin user. Then log out and never use root again.
2 Create IAM Admin User (your daily-use account) ▾

Go to AWS Console → IAM → Users → Create User

FieldValue
Usernameorbitumai-admin
Access type✅ AWS Management Console access
PermissionsAttach directly → AdministratorAccess
Console passwordCustom — save this securely

Also create a programmatic access key for CLI/Cursor use:

  • Click your user → Security credentials tab → Create access key
  • Choose "CLI" as use case
  • Download the .csv file — this contains your Access Key ID + Secret. Save it to a password manager.
3 Install & Configure AWS CLI on Windows ▾
PowerShell — Install AWS CLI
# Option 1: via winget (recommended)
winget install -e --id Amazon.AWSCLI

# Option 2: Download MSI from https://aws.amazon.com/cli/
# Then verify install
aws --version
# Expected: aws-cli/2.x.x
Configure CLI with your IAM credentials
aws configure

# You will be prompted for:
AWS Access Key ID: AKIAIOSFODNN7EXAMPLE        # from .csv file
AWS Secret Access Key: wJalrXUtnFEMI/K7MDENG   # from .csv file
Default region name: us-east-1                  # closest to Texas
Default output format: json

# Test it works
aws sts get-caller-identity
✅
If you see your Account ID and UserID, your CLI is configured correctly. You're ready for EC2.

EC2 Instance Launch

Launch your Ubuntu server on AWS — same OS as your Hostinger VPS (Ubuntu 24.04)

💡
Think of EC2 as your VPS replacement. It's the same concept as your Hostinger VPS at IP 31.220.18.167, but hosted on AWS with much better scaling, snapshots, and tooling.
1 Launch EC2 via AWS Console (easiest for beginners) ▾

Go to AWS Console → EC2 → Launch Instance

SettingValueWhy
Nameorbitumai-prodYour server name
OSUbuntu Server 24.04 LTSSame as Hostinger
Instance typet3.medium (2 vCPU, 4GB)Handles 4-6 apps
Key pairCreate new → orbitumai-key → Download .pemSSH access
Storage40 GB gp3Enough for Docker images
Security GroupCreate new (see next step)Firewall rules
2 Security Group — Open Required Ports ▾

While creating the instance, under Network Settings → Create Security Group, add these inbound rules:

PortProtocolSourcePurpose
22TCPMy IP onlySSH access
80TCPAnywhere (0.0.0.0/0)HTTP
443TCPAnywhereHTTPS
81TCPMy IP onlyNginx Proxy Admin
9000TCPMy IP onlyPortainer
5678TCPAnywheren8n
⚠️
Set SSH port 22 to "My IP", not Anywhere. This blocks all brute-force attacks from the internet.
3 SSH Into Your EC2 from Windows ▾

After the instance starts, copy the Public IPv4 address from EC2 dashboard.

Windows — Git Bash or PowerShell
# Fix key permissions first (Git Bash)
chmod 400 orbitumai-key.pem

# Connect to server
ssh -i "orbitumai-key.pem" ubuntu@YOUR_EC2_PUBLIC_IP

# Example:
ssh -i "orbitumai-key.pem" ubuntu@54.123.45.67
First-time server setup (run after SSH)
# Update system packages
sudo apt update && sudo apt upgrade -y

# Install essentials
sudo apt install -y curl git nano ufw fail2ban

# Setup firewall
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable

# Create app directories (mirror your Hostinger structure)
mkdir -p ~/apps/{orbitumai,blogtly,pix60,n8n,openclaw}
mkdir -p ~/proxy/{data,letsencrypt}
4 Assign an Elastic IP (Static IP — won't change on reboot) ▾
  • Go to EC2 → Elastic IPs → Allocate Elastic IP address
  • Click Allocate, then Associate Elastic IP → select your instance
  • This gives you a permanent IP like 54.xx.xx.xx — use this for all DNS records
  • Free when attached to a running instance. ~$0.005/hr if instance is stopped.
✅
This is equivalent to your Hostinger static IP. Point all your domain DNS A-records to this Elastic IP.

Nginx Proxy Manager on AWS

Route all domains to containers — same as proxy.orbitumai.com on Hostinger

1 Deploy NPM with docker-compose.yml ▾
~/proxy/docker-compose.yml
version: '3.8'
services:
  npm:
    image: jc21/nginx-proxy-manager:latest
    container_name: nginx-proxy-manager
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
      - "81:81"       # Admin UI
    volumes:
      - ./data:/data
      - ./letsencrypt:/etc/letsencrypt
    networks:
      - proxy

networks:
  proxy:
    external: true   # Use the network we created above
Start NPM
cd ~/proxy
docker compose up -d

# Default credentials: admin@example.com / changeme
# Access at: http://YOUR_EC2_IP:81
2 Add a Proxy Host (route domain → container) ▾

In NPM Admin → Proxy Hosts → Add Proxy Host:

FieldValue
Domain Namesapp.orbitumai.com
Forward Hostnamecontainer-name (NOT the IP!)
Forward PortApp's internal port e.g. 3000
SSL CertificateRequest new → Let's Encrypt → Force SSL ✅
Block Common Exploits✅ Enable
⚠️
Always use container name, not IP as Forward Hostname. Example: orbitumai-app, not 172.17.0.2. This is what fixed your n8n 502 errors.

ENV Variables — The Right Way on AWS

Store secrets securely — never hardcode in code or commit to GitHub

🚨
Never put API keys or DB passwords directly in docker-compose.yml or .env files committed to GitHub. Use the methods below instead.
1 Method 1 — .env file on server (simplest, good for solo projects) ▾
Create .env file on EC2 (NOT in GitHub)
# SSH into EC2, then:
nano ~/apps/orbitumai/.env

# Add your variables:
NEXT_PUBLIC_SUPABASE_URL=https://xxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOi...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOi...
ANTHROPIC_API_KEY=sk-ant-...
DATABASE_URL=postgresql://user:pass@host:5432/db
NEXTAUTH_SECRET=your-random-secret-here
NEXTAUTH_URL=https://orbitumai.com

# Save: Ctrl+X → Y → Enter
Reference .env in docker-compose.yml
services:
  orbitumai-app:
    image: orbitumai-app:latest
    container_name: orbitumai-app
    env_file:
      - .env             # loads all variables from .env
    ports:
      - "3000:3000"
    networks:
      - proxy

Add .env to your .gitignore to ensure it never gets committed:

.gitignore
.env
.env.local
.env.production
*.pem
node_modules/
2 Method 2 — AWS Systems Manager Parameter Store (most secure) ▾

Store secrets in AWS, pull them at deploy time. Best for team projects.

Store a secret in AWS SSM
# Store a secret (run from your Windows machine)
aws ssm put-parameter \
  --name "/orbitumai/prod/ANTHROPIC_API_KEY" \
  --value "sk-ant-your-key-here" \
  --type SecureString

aws ssm put-parameter \
  --name "/orbitumai/prod/SUPABASE_SERVICE_ROLE_KEY" \
  --value "eyJhbGci..." \
  --type SecureString

# Pull secrets to .env file on EC2 at deploy time
aws ssm get-parameter \
  --name "/orbitumai/prod/ANTHROPIC_API_KEY" \
  --with-decryption \
  --query "Parameter.Value" \
  --output text

Migrate from Vercel to AWS

          
Move your Next.js apps from Vercel to Docker on AWS EC2

        

      


      
💡
Vercel is serverless. AWS EC2 is a persistent server. Your Next.js apps will run as Docker containers using next start — the output is identical, you just self-host it.


      

        

          1
          Dockerfile for Next.js Apps (works for all your apps)
          ▾
        

        

          

            
Dockerfile — place in Next.js app root

            
FROM node:20-alpine AS base

# Install dependencies
FROM base AS deps
WORKDIR /app
COPY package.json package-lock.json* ./
RUN npm ci

# Build
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

# Production image
FROM base AS runner
WORKDIR /app
ENV NODE_ENV production
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs
COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

USER nextjs
EXPOSE 3000
ENV PORT 3000
CMD ["node", "server.js"]

          

          
Also add to your next.config.js:

          

            
next.config.js — enable standalone output

            
module.exports = {
  output: 'standalone',   // Required for Docker
}

          

        

      


      

        

          2

          ▾
        

        

          
    
            
  • In Vercel: go to Project → Settings → Environment Variables

  • Copy all variables listed there
    • 
            
  • On EC2: create a .env file with those same key=value pairs

  • Reference the .env file in your docker-compose.yml via env_file: .env

NEXT_PUBLIC_ variables must be present at build time, not just runtime. Include them in your Dockerfile's build stage using ARG and ENV, or pass them via GitHub Actions secrets during the build step.

    •
